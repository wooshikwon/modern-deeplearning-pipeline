## 핵심 발견

### (1) 문서 의도 우선 — 코드를 문서에 맞춰야 하는 곳

**1-1. Mini-epoch NaN 발생 시 나머지 mini-epoch을 건너뛰지 않음**

spec-rl은 "NaN in mini-epoch: backward_and_step returns None → skip all remaining mini-epochs, global_step not incremented"라고 명시한다. 현재 코드는 NaN이 발생해도 `for` 루프를 계속 돌며 나머지 mini-epoch을 실행한다. NaN 상태에서 후속 forward/backward를 시도하면 그래디언트 오염 및 불필요한 연산이 발생한다. `all_nan` 플래그로 global_step 증가는 올바르게 차단되지만, NaN 발생 시 즉시 break해야 한다.

- 코드 위치: `mdp/training/rl_trainer.py:795-819`
- 수정 방향: `result is None`일 때 루프 내에서 `break`를 추가하여 나머지 mini-epoch을 즉시 중단

**1-2. DualEncoderHead `embed_dim` 파라미터와 catalog YAML `hidden_dim` 키 불일치**

`DualEncoderHead.__init__`은 `embed_dim`을 첫 번째 인자로 받지만, catalog YAML(`clip-vit-base.yaml`)의 `default_head` 섹션은 `hidden_dim: 512`를 전달한다. ComponentResolver가 catalog의 `default_head`를 resolve하면 `DualEncoderHead(hidden_dim=512)`가 호출되어 `TypeError: unexpected keyword argument 'hidden_dim'`이 발생한다.

- 코드 위치: `mdp/models/heads/dual_encoder.py:22`, `mdp/models/catalog/feature_extraction/clip-vit-base.yaml:17`
- 수정 방향: catalog YAML의 `hidden_dim`을 `embed_dim`으로 변경

**1-3. Validation fallback chain에서 training_step fallback 누락**

spec-training은 validation fallback의 마지막 단계로 "`model.training_step(batch)`로의 최종 fallback"을 명시한다. 코드의 `_validate_fallback`은 `outputs["loss"]` → `outputs.loss` → `loss_fn(logits, labels)` 순서로 시도하지만, 모든 경로가 실패하면 빈 dict를 반환한다. `model.training_step(batch)` 호출이 구현되어 있지 않아 validation 메트릭이 유실될 수 있다.

- 코드 위치: `mdp/training/trainer.py:510-533`
- 수정 방향: `_validate_fallback` 마지막에 `model.training_step(batch)` 호출을 추가하여 fallback chain을 완성

**1-4. RL `create_models()`에 `skip_base_check` 파라미터 미지원**

spec-model은 RL `create_models()`도 SFT `create_model()`과 동일한 5단계 조립을 거친다고 명시한다. `skip_base_check`는 MLflow artifact에서 모델을 복원할 때 BaseModel 검증을 건너뛰는 파라미터인데, `create_models()`에는 이 파라미터가 없어 `_assemble_model()`에 항상 `False`가 전달된다. RL 모델의 artifact 복원 시나리오에서 BaseModel이 아닌 모델(HF pretrained 등)을 로드하면 `TypeError`가 발생할 수 있다.

- 코드 위치: `mdp/factory/factory.py:262-277`
- 수정 방향: `create_models(skip_base_check: bool = False)` 파라미터를 추가하고 `_assemble_model()`에 전달

**1-5. TrainResult에 `run_id` 필드 누락**

spec-infra는 MLflow를 단일 진실 원천으로 설계하며, 추론/서빙/내보내기가 모두 `--run-id`를 참조한다. `InferenceResult`에는 `run_id`가 포함되지만, `TrainResult`에는 없다. 에이전트가 학습 후 자동으로 추론/서빙을 연결하려면 학습 결과에서 `run_id`를 받아야 한다.

- 코드 위치: `mdp/cli/schemas.py:18-37`, `mdp/cli/train.py:109-122`
- 수정 방향: `TrainResult`에 `run_id: str | None = None` 필드를 추가하고, train 완료 후 MLflow run_id를 결과에 포함

**1-6. 메모리 추정 전략 추천 threshold가 `<` 대신 `<=`여야 함**

spec-infra는 "모델이 VRAM의 70% 이내 → ddp, 2배 이내 → fsdp"로 경계값을 `<=`로 명시한다. 코드는 `model_mem_gb < gpu_vram * 0.7`과 `model_mem_gb < gpu_vram * 2`로 strict `<`를 사용하여, 정확히 70%나 2배일 때 한 단계 상위 전략으로 에스컬레이션된다.

- 코드 위치: `mdp/utils/estimator.py:266,270`
- 수정 방향: `<`를 `<=`로 변경하여 경계값에서 불필요한 에스컬레이션 방지

**1-7. task_fields의 tokenizer/augmentation 필요성 판단이 fields role 기반이 아닌 task name 기반**

spec-foundation의 BusinessValidator는 "tokenizer/augmentation necessity derived from data.fields roles (text→tokenizer required, image→augmentation needed), NOT from task name"을 원칙으로 명시한다. 현재 코드는 `validate_task_fields(recipe.task, ...)`를 호출하여 `TASK_PRESETS[task].required_config`로 판단하는데, 이는 **task name 기반**이다. fields 역할 기반의 역방향 검증(text가 없는데 tokenizer가 있으면 warning)은 별도로 구현되어 있지만, 정방향(text가 있는데 tokenizer가 없으면 warning)은 task 기반으로만 동작한다.

- 코드 위치: `mdp/settings/validation/business_validator.py:244-259`, `mdp/task_taxonomy.py:105-109`
- 수정 방향: `required_config` 판정을 fields의 role 기반으로 전환 — fields에 text 역할이 있으면 tokenizer warning, image 역할이 있으면 augmentation warning. task_preset 기반 경고는 보조 수단으로 남기거나 제거

### (2) 현재 구현 우선 — 문서를 코드에 맞춰야 하는 곳

**2-1. Expert Parallelism이 strategy setup 전에 적용됨**

spec-training은 "Expert Parallelism applied after distributed wrapping"이라고 명시하지만, 코드에서는 EP가 strategy.setup() **전에** 적용된다. EP는 process group이 필요하므로 먼저 `init_process_group`을 수행하고 EP를 설치한 뒤, strategy가 wrapping하는 흐름이 기술적으로 올바르다. 코드 내부의 docstring(`moe.py`)도 이 순서를 확인한다.

- 코드 위치: `mdp/training/trainer.py:244-258`
- 수정 방향: spec-training을 "Expert Parallelism applied before distributed wrapping (requires process group initialization first)"로 갱신

**2-2. backward_and_step의 scheduler 호출이 caller 책임**

spec-training은 "Step-level scheduler: called inside backward_and_step when interval=='step'"이라고 명시하지만, `backward_and_step` 내부에는 interval 판별 로직이 없다. 대신 Trainer가 `step_schedulers` dict를 조건부로 구성하여 전달한다 (`scheduler_interval == "step"`일 때만 dict에 포함). 이 패턴은 `backward_and_step`을 SFT/RL 양쪽에서 재사용 가능하게 하므로 현재 구현이 더 유연하다.

- 코드 위치: `mdp/training/_common.py:127-129`, `mdp/training/trainer.py:418-422`
- 수정 방향: spec-training을 "caller가 step-level scheduler만 schedulers dict에 포함하여 전달" 방식으로 갱신

**2-3. RL Algorithm 인터페이스가 `compute_loss()`가 아닌 `__call__()`**

spec-rl은 "All implement compute_loss() interface"라고 명시하지만, DPOLoss, GRPOLoss, PPOLoss 모두 `__call__()` 메서드로 호출된다. `compute_loss()`라는 명시적 메서드는 존재하지 않는다. `__call__`은 Python의 callable convention에 부합하며, RLTrainer에서 `self.algorithm(trainable_out, frozen_out, batch)`로 자연스럽게 호출된다.

- 코드 위치: `mdp/training/losses/rl.py:83,206,281`
- 수정 방향: spec-rl의 "compute_loss()" 표현을 "`__call__(trainable_out, frozen_out, batch) → dict[str, Tensor]`"로 갱신

**2-4. BusinessValidator가 6개 규칙 (rl_models 추가)**

spec-foundation은 BusinessValidator를 5개 규칙(head_task, adapter, distributed_batch, task_fields, label_strategy_fields)으로 명시하지만, 실제 코드에는 `rl_models`가 6번째 체크로 추가되어 있다. RL recipe의 per-model head/adapter 검증을 담당하며, 합리적인 추가다.

- 코드 위치: `mdp/settings/validation/business_validator.py:42-49`
- 수정 방향: spec-foundation에 `rl_models` 규칙을 추가하여 6개 규칙으로 업데이트

### (3) 논의 필요 — 판단 유보

**3-1. `label_strategy` default "none"의 의미와 명시 선언 원칙**

spec-schema는 "label_strategy는 명시적 필드 — 자동 파생 불가"를 원칙으로 삼는다. 현재 `label_strategy: str = "none"`으로 default가 있어 사용자가 생략해도 schema validation을 통과한다. `"none"`일 때는 `_check_label_strategy_fields`에서 required_fields가 빈 리스트이므로 fields 검증도 발생하지 않는다.

두 가지 해석이 가능하다:
- (A) `"none"`이 합법적인 기본값이므로 "자동 파생이 없다"는 의미로 충분. `mdp init`이 task 기반 기본값을 생성하므로 대부분의 사용자는 명시적으로 설정하게 된다.
- (B) default를 제거하여 반드시 사용자가 값을 제공하도록 강제. 실수로 label_strategy를 빼먹는 케이스를 방지.

만약 (B)를 채택하면 기존 YAML에 label_strategy가 없는 경우 즉시 ValidationError가 발생한다. (A)를 채택하면 spec 문구를 "default: none, no auto-derivation"으로 명확히 한다.

- 코드 위치: `mdp/settings/schema.py:65`
- 탐색 질문: vision-only 워크플로우(image classification 등)에서 label_strategy를 매번 명시하는 것이 사용자 경험에 어떤 영향을 미치는가? `mdp init`이 생성하는 기본 recipe에는 항상 label_strategy가 포함되는가?

**3-2. Generation path에서 on_batch_end 콜백이 grad_accum_steps에 의해 누락될 수 있음**

RLTrainer의 generation path는 `force_step=True`로 매 mini-epoch마다 optimizer step을 실행하므로 gradient accumulation과 무관하다. 그러나 `on_batch_end` 콜백은 `batch_idx % self.grad_accum_steps == 0` 조건에서만 발화된다. `grad_accum_steps > 1`일 때 일부 generation step에서 ModelCheckpoint의 `every_n_steps` 기반 저장이나 LRMonitor 업데이트가 누락될 수 있다.

offline path(DPO)는 표준 gradient accumulation을 사용하므로 이 조건이 적절하다. Generation path와 offline path의 콜백 발화 조건을 분리할 필요가 있는지 논의가 필요하다.

- 코드 위치: `mdp/training/rl_trainer.py:647`
- 탐색 질문: Generation path에서 grad_accum_steps > 1을 설정하는 실제 사용 시나리오가 있는가? 있다면 on_batch_end를 항상 발화하는 것이 맞고, 없다면 grad_accum_steps=1을 generation path에서 강제하는 방법도 있다.

**3-3. SFT Trainer on_batch_end가 에폭 마지막 배치에서 누락될 수 있음**

SFT Trainer의 `on_batch_end`는 `(step + 1) % self.grad_accum_steps == 0` 조건에서만 발화된다. 에폭의 총 배치 수가 `grad_accum_steps`의 배수가 아닌 경우, 마지막 배치에서 콜백이 발화되지 않는다. 이와 연관하여 `backward_and_step`에 `force_step`이 정의되어 있지만 SFT Trainer에서는 사용하지 않아, 에폭 마지막의 잔여 gradient도 손실된다.

- 코드 위치: `mdp/training/trainer.py:443`, `mdp/training/_common.py:98`
- 탐색 질문: 에폭 마지막에 잔여 gradient를 flush하는 것이 학습 품질에 미치는 영향은? 많은 프레임워크(PyTorch Lightning 등)는 에폭 마지막에 force step을 수행한다. 이를 채택하면 on_batch_end도 자연스럽게 발화된다.

### (4) /coding 위반

**4-1. QLoRA + head 설정이 spec의 "forbidden"과 달리 warning으로만 처리됨**

spec-model은 "QLoRA head attachment forbidden"이라고 명시하지만, 코드는 `logger.warning()`만 발생시키고 head_config를 무시한 채 진행한다. 사용자가 실수로 QLoRA recipe에 head를 설정해도 조용히 무시되어 의도하지 않은 모델 구조로 학습될 수 있다.

- 코드 위치: `mdp/factory/factory.py:77-81`
- 수정 방향: `ValueError`를 발생시켜 "forbidden" 의미에 맞추거나, spec을 "warning + ignore"로 완화. BusinessValidator에서 "QLoRA + head" 조합을 error로 잡는 것이 더 적절 (학습 진입 전 차단)

**4-2. Augmentation DSL에서 dot-path custom transform이 미구현**

spec-data는 "no dot → torchvision.transforms.v2 lookup"이라는 조건을 명시하여, dot이 포함된 경우 커스텀 transform의 Python import path 해석을 암시한다. `_component_` 패턴의 ComponentResolver와 동일한 방식이 기대되지만, 현재 코드는 `getattr(v2, transform_type, None)`만 시도하고 dot-path를 처리하지 않는다.

- 코드 위치: `mdp/data/transforms.py:65-69`
- 수정 방향: dot이 포함된 `transform_type`은 ComponentResolver와 동일하게 `importlib`로 동적 import 처리

**4-3. Tokenizer docstring의 YAML 예시에 `padding: "max_length"`가 있어 설계 원칙과 모순**

spec-data는 "항상 padding=False (DataCollator가 동적 패딩 담당)"을 원칙으로 명시하며, 코드의 `build_tokenizer` 함수도 `padding` 기본값을 `False`로 설정한다. 그러나 함수 docstring의 YAML 예시에 `padding: "max_length"`가 포함되어 있어, 사용자가 이를 따르면 불필요한 고정 길이 패딩이 적용된다.

- 코드 위치: `mdp/data/tokenizer.py:36`
- 수정 방향: docstring YAML 예시에서 `padding: "max_length"` 줄을 제거

**4-4. ModelCheckpoint의 epoch 저장이 on_validation_end와 on_batch_end에서 불일치**

`on_validation_end`에서는 `epoch + 1`을 저장하고(이미 완료된 epoch 다음부터 resume), `on_batch_end`에서는 kwargs의 `epoch`를 그대로 저장한다. resume 시 validation-end checkpoint은 다음 epoch부터 시작하지만, batch-end checkpoint은 같은 epoch의 `step_in_epoch`부터 시작한다. 두 checkpoint 유형의 epoch 의미가 다르면 resume 로직에서 혼동이 생길 수 있다.

- 코드 위치: `mdp/training/callbacks/checkpoint.py:255` (epoch+1) vs `mdp/training/callbacks/checkpoint.py:217` (epoch 그대로)
- 수정 방향: epoch 저장 기준을 통일. on_batch_end에서는 `step_in_epoch`이 있으므로 같은 epoch을 저장하는 것이 맞고, on_validation_end에서도 현재 epoch을 저장한 뒤 resume 로직에서 validation 완료 여부를 판단하는 것이 더 명확

**4-5. weighted-NTP가 docstring에 언급되지만 미구현**

`losses/rl.py`의 모듈 docstring에 "DPO, weighted-NTP, GRPO, PPO"가 나열되지만, weighted-NTP 클래스는 구현되어 있지 않다. spec-rl은 이를 "custom algorithm example"로 기술하지만, docstring이 마치 구현된 것처럼 나열하여 혼동을 줄 수 있다.

- 코드 위치: `mdp/training/losses/rl.py:1`
- 수정 방향: docstring에서 weighted-NTP를 제거하거나, "커스텀 알고리즘 예시로 구현 예정" 표기

**4-6. _forward_model의 preference/causal 분기가 상호 배타적이지 않음**

`_forward_model`에서 `chosen_input_ids`와 `input_ids` 체크가 독립적인 `if` 문으로 되어 있어, 배치에 두 키가 모두 존재하면 양쪽 분기가 모두 실행된다. 현재 PreferenceCollator 출력에는 `input_ids`가 없으므로 실제 문제가 발생하지 않지만, 방어적 코딩 관점에서 `elif`로 상호 배타적으로 처리하는 것이 안전하다.

- 코드 위치: `mdp/training/rl_trainer.py:881,893`
- 수정 방향: `if "input_ids" in batch:`를 `elif "input_ids" in batch:`로 변경하여 preference 형태가 우선되도록 보장
