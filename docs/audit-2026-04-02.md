### 핵심 철학 요약

MDP는 MMP(Modern ML Pipeline)의 Config-driven 설계를 딥러닝으로 확장한 CLI 도구다. 9개 핵심 원칙이 전체 설계를 관통한다:

1. **Config-driven DL** — YAML만으로 학습/추론/서빙
2. **`_component_` 패턴 통일** — Registry 대신 Python import path 직접 매핑
3. **Recipe/Config 2-File** — 실험 정의와 인프라 설정 분리
4. **Task는 검증/프리셋만** — 실제 동작은 `fields` + `head` + `model`이 결정
5. **3-Tier 단방향 의존** — import-linter로 경계 강제
6. **조기 실패 + 일괄 보고** — 4단계 검증, 에러 수집 후 단일 ValueError
7. **dict I/O** — 모달리티 불문 dict 인터페이스
8. **프로세스 내 복구 없음** — 체크포인트 + 클린 재시작
9. **비핵심 실패 격리** — MLflow/monitoring 실패는 warning만

## 문서 간 모순

코드 감사에 앞서, 설계 문서 자체에서 발견된 문서 간 충돌을 정리한다. 이들은 코드가 어떤 해석을 따랐는지와 무관하게 문서 자체가 해결해야 할 문제다.

### 1. loss 생략 기준 불일치 (02 vs 03)

02-schema-specification은 loss 생략 조건을 "HuggingFace 모델이 labels를 받으면"으로 한정한다. 그러나 03-schema-examples는 YOLO(비-HuggingFace 모델)도 loss를 생략하는 사례를 제시하며 "training_step 구현 모델"이라는 새로운 카테고리로 설명한다. 실질 기준은 "모델이 내부적으로 loss를 계산하는가"이므로, 02의 기술이 과도하게 좁다. **02를 "training_step이 스칼라 loss를 반환하는 모델"로 확장해야 한다.**

### 2. 스펙 미완성 영역의 예시 선행 (02 vs 03)

`modules_to_save`, FSDP 샤딩 전략(`SHARD_GRAD_OP`) 등 구현에 필요한 필드가 02 스펙에 없고 03 예시에서만 등장한다. 스펙이 예시보다 덜 완성된 상태다. **02에 누락 필드를 보충해야 한다.**

### 3. Triton 서빙 공백 (08 vs usage-scenarios)

08은 ONNX 실시간 서빙을 "Triton 별도"로 권장하지만 Triton은 "설계 예정". usage-scenarios는 ONNX를 서빙 O로 분류. 세 문서를 합치면 ONNX 실시간 서빙의 권장 경로가 사실상 없다. **usage-scenarios의 ONNX 서빙 O 표기를 수정하거나, ONNX 실시간 서빙의 대안 경로를 명시해야 한다.**

### 4. 에러 스키마 3중 분산 (08, 09, AGENT.md)

에러 타입 정의, TrainResult/InferenceResult/EstimateResult 구조가 08, 09, AGENT.md에 분산 기술되어 있다. 09는 "AGENT.md가 권위 원천이므로 중복하지 않는다"고 선언하면서도 동일 내용을 기술한다. **AGENT.md를 단일 원천으로 확정하고 08/09의 중복을 제거해야 한다.**

### 논의 필요

코드 흐름을 재확인한 결과, 원래 12건 중 일부는 관점 자체가 틀렸다. 재분류하여 정리한다.

#### 실제로 논의가 필요한 항목 (3건)

**1. QLoRA에서 head 무시 시 경고 없음**

Factory의 `_build_model`은 두 갈래로 나뉜다. adapter가 QLoRA가 아니면 일반 흐름(pretrained 로딩 → head 교체 → adapter 적용)을 타고, QLoRA이면 별도 메서드 `_build_qlora_model`로 빠진다. 이 QLoRA 전용 분기는 양자화 로딩과 LoRA 적용을 한 번에 처리하는데, **head 부착 단계가 아예 없다**. Recipe에 `head` 섹션이 있어도 무시된다.

왜 head를 적용할 수 없는가: QLoRA는 양자화 로딩과 adapter 적용을 한 번에 처리하므로, 그 사이에 head를 교체할 단계가 없다. `AutoModelForCausalLM` 등 head가 내장된 모델을 양자화 로딩하는 것이 QLoRA의 전제다. Recipe에 `head`를 적었는데 아무 피드백 없이 무시되면 사용자는 head가 적용된 줄 안다.

**권장**: `_build_qlora_model` 진입부에서 `recipe.head is not None`이면 `logger.warning("QLoRA는 head가 내장된 모델(AutoModelFor* 등)만 지원합니다. Recipe의 head 설정은 적용되지 않습니다.")`를 남긴다.

**2. apply_adapter의 model: Optional 시그니처**

`apply_adapter(model, adapter_config)` 함수는 lora/prefix_tuning에서는 반드시 model을 받고, qlora에서는 반드시 None을 받는다. 하나의 함수가 두 가지 근본적으로 다른 흐름을 처리하고 있다.

이 설계가 존재하는 이유: adapter 적용은 "진입점 하나로 method를 라우팅" 패턴을 따른다. Factory가 `apply_adapter` 하나만 알면 되고, 새 adapter method가 추가되어도 Factory를 수정할 필요가 없다. 이 단순성은 가치가 있다.

긴장 관계: `model: nn.Module | None` 시그니처는 caller에게 "model 없이도 호출 가능"이라는 잘못된 신호를 준다. lora나 prefix_tuning에서 None을 넘기면 PEFT 내부에서 런타임 에러가 발생하는데, 이 에러 메시지가 MDP와 무관하여 디버깅이 어렵다.

**권장**: 현재 구조를 유지하되, lora/prefix_tuning 분기 진입부에서 `if model is None: raise ValueError("lora/prefix_tuning에는 model이 필요합니다")` 가드를 추가한다. Factory 쪽은 이미 QLoRA만 None을 넘기므로 동작 변경 없음.

**3. val 데이터를 명시적으로 지정하고, `random_split` fallback을 제거한다**

현재 `create_dataloaders`에서 val split이 없으면 `random_split`으로 train에서 80/20 분리한다. 이 시점에서 train augmentation이 이미 적용되어 있어 val 데이터에도 random crop, flip 등이 걸린다. 잘못된 val 메트릭으로 모델을 선택하면 과적합을 놓치므로, **자동 split이 없는 것보다 위험하다.**

**권장**:

1. **`DataSpec`에 `val_source` 필드를 추가하여 val 데이터를 명시적으로 지정**:
```yaml
data:
  source: "./train.csv"
  val_source: "./val.csv"       # 명시적 지정. 없으면 validation 미수행
```
또는 기존 `data_files` 활용:
```yaml
data:
  source: "./data"
  data_files:
    train: "train.csv"
    validation: "val.csv"
```

2. **Recipe 파싱 시점에 조기 검증**:
   - `val_source`가 지정됐지만 파일이 존재하지 않으면 → 즉시 에러
   - `val_source`가 지정됐지만 train과 컬럼 구조가 다르면 → 즉시 에러
   - `val_source`가 없으면 → warning 후 validation 없이 학습

3. **`random_split` fallback 제거**. validation은 학습에 필수가 아니다. 고정 epoch 학습 후 `mdp inference --metrics`로 평가하는 것도 유효한 흐름이다.

**~~4. 분산 학습 결과 미수집~~** → 여전히 미수정이나, MLflow에 metric이 기록되므로 `--run-id`로 결과를 조회할 수 있다. CLI JSON 출력이 빈 값인 것은 UX 문제이며 우선순위 낮음.

#### 관점 재조정 — 문제가 아닌 항목 (5건)

**~~1. Recipe의 training 섹션이 추론에서도 필수~~** → 해소됨

serve/inference 재설계로 사용자가 Recipe를 직접 지정하지 않게 되었다. `--run-id`로 MLflow artifact를 참조하며, artifact 내 `recipe.yaml`은 학습 시점에 이미 완성된 상태이므로 training 섹션이 포함되어 있다. "추론 전용 Recipe"라는 개념 자체가 불필요해졌다.

**~~5. configure_optimizers 반환 dict 구조 미정의~~** → 코드에 이미 정의됨

`Trainer._create_optimizer`(trainer.py:111-118)가 이 dict를 소비하는 유일한 코드이며, `custom["optimizer"]` 키를 꺼낸다. 즉 구조는 `{"optimizer": Optimizer}`. 타입을 좁히면 좋겠지만, 이 메서드를 오버라이드하는 사용자가 PyTorch Lightning의 `configure_optimizers` 관례(`{"optimizer": ..., "lr_scheduler": ...}`)를 따를 수 있으므로, 지금 단계에서 TypedDict로 강제하면 유연성을 잃는다. BaseModel의 docstring에 기대 구조를 명시하는 정도면 충분하다.

**~~10. on_batch_end에 epoch kwarg 추가 전달~~** → 의도된 동작

BaseCallback의 모든 hook은 `**kwargs`를 받는 설계다. Trainer가 콜백에 부가 정보(`epoch`, `global_step`, `model`, `optimizer` 등)를 kwargs로 전달하는 것은 콜백이 필요한 정보에 접근하게 하려는 의도다. 실제로 `ModelCheckpoint.on_batch_end`는 `kwargs.get("global_step")`, `kwargs.get("model")` 등을 사용한다. 문서의 `on_batch_end(step, metrics)` 시그니처는 필수 인자만 나열한 것이고, kwargs는 확장 포인트이므로 불일치가 아니다.

**~~11. checkpoint epoch+1 불일치~~** → resume 정합성을 위한 의도적 설계

`on_validation_end`에서 `epoch + 1`을 저장하는 이유: 이 체크포인트는 "epoch N이 완료된 상태"를 기록하므로, resume 시 `start_epoch = state["epoch"]`로 **다음 epoch부터** 시작해야 한다. 만약 `epoch`(0-indexed)를 그대로 저장하면 resume 시 완료된 epoch를 다시 학습하게 된다. 반면 `on_batch_end`의 step-level 체크포인트는 "이 epoch의 중간 상태"이므로 현재 epoch 값(`epoch`)이 맞다 — resume 시 같은 epoch를 이어서 학습해야 하기 때문이다. 즉 두 케이스에서 epoch의 의미가 다른 것은 **의도된 설계**이며, `trainer_state.json`의 epoch 필드는 "resume 시작 epoch"으로 해석하면 일관된다.

**~~12. estimator 전략 추천 문서 표현 모호~~** → 문서 표현 문제 (코드 이슈 아님)

코드는 명확하다: `model_mem_gb < gpu_vram * 0.7`이면 ddp. 문서의 "VRAM 70%"는 같은 의미를 다소 축약한 것이다. 코드가 맞으므로 "현재 구현 우선 — 문서 갱신" 섹션으로 이관한다.

#### 버그로 재분류 (3건)

**~~7. image 컬럼 keep vs pixel_values 변환 시점~~** → 의도된 방어적 설계 (이슈 아님)

코드 흐름을 추적하면: HuggingFace `imagefolder`로 로드한 데이터셋은 컬럼명이 `"image"`이다. `_rename_columns`에서 fields 매핑으로 리네임할 수 있지만, 대부분의 경우 `fields: {image: image}`로 지정하므로 컬럼명은 `"image"` 그대로다. `set_transform` 콜백이 `"image"` 키에 vision transform을 적용하고, 모델의 `forward`가 `batch["image"]`를 `batch["pixel_values"]`로 변환하는 것은 모델 구현의 책임이다. keep 집합에 둘 다 포함하는 것은 "어떤 이름이든 삭제되지 않도록" 하는 방어적 설계이며, 혼란이 아니라 안전장치다.

**~~8. baseline_info 미초기화~~** → **수정 완료**

try 블록 앞에 `baseline_info = None`을 추가하여 해결.

**~~9. for_estimation이 Config를 받지 않는 구조~~** → 현재 설계가 적절 (재분류)

`mdp estimate`의 목적은 "학습 시작 전에 대략적인 GPU 메모리를 예측"하는 것이다. estimator.py는 모델 파라미터 수에서 역산하는 휴리스틱을 사용하며, 기본 GPU VRAM을 24GB(RTX 4090/3090)로 가정한다. 이 도구의 가치는 "Recipe만으로 빠르게 감을 잡는 것"이지, 정밀한 프로파일링이 아니다. Config까지 요구하면 `mdp estimate recipe.yaml`의 간결함이 사라진다. 실제 환경에서 정밀 추정이 필요하면 `mdp train --dry-run` 같은 별도 경로가 적절하며, estimate가 Config를 받지 않는 것은 의도된 트레이드오프다.
