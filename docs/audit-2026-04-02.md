### 핵심 철학 요약

1. **선언형 YAML 파이프라인**: Recipe(실험 정의) + Config(인프라 설정) 분리로 재현성 보장
2. **유연한 이원화**: `_component_` 패턴(optimizer/scheduler/loss/head/callback) vs 고정 스키마(model/data/adapter/training)

4. **점진적 분산 학습**: 단일 GPU → DDP → FSDP → DeepSpeed → MoE Expert Parallelism
5. **검증 기반 안전성**: 3단 validators (Business/Catalog/Compat)
6. **Forward 계약**: loss 지정 시 `dict(logits=...)`, 미지정 시 `training_step() → scalar`

### 논의 필요 — 판단 유보

#### ~~1. MoE `ep_forward`의 `top_k` 하드코딩~~ **수정 완료**

- **위치**: `training/strategies/moe.py`
- **수정**: `_detect_top_k(model)`이 `config.num_experts_per_tok`에서 자동 추출. 미감지 시 기본값 2.

#### 2. Seq2SeqLMHead vs CausalLMHead 중복

- **위치**: `models/heads/causal_lm.py:20-35` vs `models/heads/seq2seq_lm.py:22-37`
- **양쪽 입장**: 구조적으로 동일 (hidden_dim → vocab_size 선형 변환). 하나는 의미론적 구분 (인코더-디코더 모델용), 하나는 구현 중복.
- **트레이드오프**: alias로 통합하면 코드 감소, 분리 유지하면 향후 독립적 확장 가능.

#### 3. DualEncoderHead의 `forward_pair()` 이중 인터페이스

- **위치**: `models/heads/dual_encoder.py:29-59`
- **양쪽 입장**: BaseHead 계약은 `forward(features) → Tensor`만 정의. `forward_pair()`는 CLIP/SigLIP 학습에 필요하지만 계약 밖.
- **트레이드오프**: BaseHead 계약을 확장하면 다른 head에 불필요한 메서드 강제. forward_pair를 별도 인터페이스로 분리 가능.

#### 4. RLTrainer validation 구현 부재

- **위치**: `training/rl_trainer.py:418-528`
- **양쪽 입장**: SFT Trainer는 `_run_validation`, `_validate` 구현. RL Trainer는 없음. `self.val_loader`가 할당되지만 사용되지 않음.
- **트레이드오프**: RL에서 validation의 의미가 SFT와 다름 (reward 기반 평가 vs metric 기반 평가). 어떤 형태의 validation이 필요한지 정의 선행.

#### ~~5. Strategy kwargs가 setup에 전달되지 않는 문제~~ **문제 아님 확인**

- **위치**: `training/trainer.py:229-234`
- **확인**: `_create_strategy()`가 `resolver.resolve({"_component_": class_path, **strategy_kwargs})`로 Strategy를 인스턴스화. kwargs는 `__init__`에서 소비되고 인스턴스 속성으로 저장됨. `setup()`은 `(model, device, optimizer)` 인터페이스만 사용. 설계 의도대로 동작.

#### 6. Estimator가 Factory와 분리됨

- **위치**: `utils/estimator.py:64-96`
- **양쪽 입장**: 파라미터 수를 class_path 문자열에서 정규식으로 추출 (heuristic). Factory의 `_load_pretrained()`로 실제 모델을 로드하면 정확하지만, 메모리 추정 목적에서 실제 로드는 과도할 수 있음.
- **트레이드오프**: 정확성 vs 속도. 현재 heuristic이 "빠른 추정"이라는 설계 의도에 부합할 수 있음.

#### 7. val split 추론 하드코딩

- **위치**: `data/dataloader.py:140-142`
- **양쪽 입장**: `_infer_val_split()`이 train → validation, 그 외 → test로 고정. 사용자가 split="val"로 지정하면 추론 실패.
- **트레이드오프**: HF datasets의 표준 split명은 "train"/"validation"/"test"이므로 대부분 커버됨. 비표준 split명은 사용자가 명시적으로 지정해야 함.

## 문서 내부 모순 (감사 중 발견)

### AGENT.md

1. **Vision-Language task 분류 불명확**: "vision_language는 별도 task가 아니다"라고 명시했으나, CLIP(`feature_extraction`)과 LLaVA(`text_generation`)의 구분 기준이 모호.

2. **Head 생략 vs 명시 규칙**: "AutoModelFor* 사용 시 head 생략 가능"이라 했으나, CLIP 예제에서 `DualEncoderHead`를 명시 지정. 언제 생략하고 언제 명시하는지 불명확.

3. **Preference prompt 선택성**: "prompt (선택)"이라 했으나, prompt 없을 때의 label masking 동작 미정의. DPO 논문에서는 prompt가 거의 필수.

### 설계 문서 간

4. **Phase 0 위상 혼재**: `plan-moe`는 Phase 1~3만 정의. `impl-moe`는 RL Trainer kwargs를 Phase 0으로 추가. 구현은 이미 완료됨 → 문서 업데이트 필요.

5. **카탈로그 `adapter_defaults.lora_with_experts` 사용 패턴 미정의**: impl-moe에서 "사용자가 명시적으로 선택"이라 했으나, Recipe의 어디서 선택하고 Factory가 어떻게 처리할지 미기술.
