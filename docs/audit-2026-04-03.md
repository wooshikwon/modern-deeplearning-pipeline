### 핵심 철학 요약

1. **Config-driven**: YAML → Pydantic → Settings → Factory → Components
2. **`_component_` 패턴**: aliases.yaml + ComponentResolver로 모든 플러그인 매핑 (Registry 제거)
3. **Recipe/Config 분리**: Recipe = 무엇을(재현성), Config = 어디서(유연성)
4. **Task ≠ 라우팅**: Task는 검증/프리셋만, 실제 동작은 `data.fields` + `head` + `model`
5. **3-Tier 아키텍처**: Atomic → Composite → Orchestrator, 단방향 의존성
6. **dict I/O**: 모달리티 독립적 인터페이스
7. **검증-먼저**: 3단 validators, 모든 에러 수집 후 단일 보고
8. **체크포인트 기반 복구**: 프로세스 내 복구 없음
9. **Agent-First**: JSON 출력, AGENT.md 단일 참조

## 핵심 발견

### 문서 의도 우선 — 코드를 문서에 맞춰야 하는 곳

#### 1. [심각] RLTrainer에서 Expert Parallelism 통합 누락

- **위치**: `training/rl_trainer.py:115, 462-479`
- **문서**: 08-implementation-infra + MoE 설계에서 EP 지원 명시
- **현재 코드**: `self.expert_parallel`이 생성만 되고 `train()` 내에서 setup/scatter/gather가 모두 빠져있다. SFT Trainer에는 완전 구현.
- **영향**: MoE 모델로 RL 학습 시 EP가 작동하지 않음
- **수정 방향**: SFT Trainer의 EP 통합 로직(setup → scatter → train → gather)을 RLTrainer에 동일하게 적용

#### 2. [심각] `on_train_end` 콜백이 MLflow 모델 export 이후에 발화됨

- **위치**: `training/trainer.py:404 vs 343-402`
- **문서**: 07에서 EMACallback은 `on_train_end`에서 EMA 가중치를 모델에 복원. MLflow summary는 `with` 블록 내 `finally`에서 실행.
- **현재 코드**: `on_train_end`가 `with mlflow_ctx:` 블록 **바깥**에서 발화됨. MLflow에 등록되는 모델 아티팩트는 EMA가 아닌 원본 가중치.
- **수정 방향**: `on_train_end`를 `_log_mlflow_summary` 이전으로 이동하거나, 모델 export를 `on_train_end` 이후로 이동

#### ~~3. `SequenceClassificationHead`가 TASK_HEAD_COMPAT에 누락~~ **재검증: 부정확**

- **정정**: `SequenceClassificationHead` 클래스는 코드베이스에 존재하지 않음. aliases.yaml에만 `ClassificationHead`를 가리키는 alias로 등록. TASK_HEAD_COMPAT에는 `ClassificationHead`가 포함되어 있으므로, 사용자가 풀 클래스 경로를 지정하면 `_extract_class_name`이 `ClassificationHead`를 추출하여 정상 검증됨. alias 이름 자체(`SequenceClassificationHead`)를 `_component_`에 직접 쓰는 경우 ComponentResolver가 해석하므로 검증 단계에서는 도달하지 않음.

#### 4. [중요] RL 관련 비즈니스 검증 부재

- **위치**: `settings/validation/business_validator.py`, `compat_validator.py`
- **문서**: 02-schema에서 3단 검증의 원칙 — 비즈니스 규칙은 validators가 담당
- **현재 코드**: RL 모델의 adapter 검증, 모델별 optimizer 일관성, RL + 분산 전략 호환성 검증이 BusinessValidator/CompatValidator에 없음. schema.py의 Pydantic validator(`check_rl_consistency`)가 기본만 처리.
- **수정 방향**: BusinessValidator에 RL 전용 검증 규칙 추가

#### 5. [중요] set_transform에서 배치 차원 미처리

- **위치**: `data/loader.py:8-16`
- **문서**: 06에서 "vision = set_transform() (lazy, 매 epoch 다른 augmentation)"
- **현재 코드**: HF `set_transform`은 slice 접근 시 `dict[str, list]`를 전달하지만, transform 람다가 개별 값에 직접 적용. DataLoader가 단일 인덱스로 접근하므로 실용적으로 동작하지만, `ds[0:5]` 같은 접근에서 오류 가능.
- **수정 방향**: 원소별 적용 보장 (리스트 입력 시 리스트 컴프리헨션)

#### 6. [보통] AdapterSpec.method 주석 불완전

- **위치**: `settings/schema.py:39`
- **문서**: 02-schema에서 adapter method = lora, qlora, prefix_tuning
- **현재 코드**: 주석이 `# "lora", "qlora"`로만 되어있고 `prefix_tuning` 누락
- **수정 방향**: 주석 수정

#### 7. [보통] AGENT.md estimate 결과 스키마 불일치

- **위치**: `cli/schemas.py:71-83` vs AGENT.md
- **문서**: AGENT.md는 중첩 구조(`model_info`, `memory_estimate`, `recommendation`)를 기술
- **현재 코드**: flat 키(`model_mem_gb`, `total_mem_gb` 등)만 사용
- **수정 방향**: 문서를 코드에 맞추거나, 코드를 중첩 구조로 확장

9. **PPO mini-epoch에서 grad_accum_steps 상호작용** (`rl_trainer.py:648-661`) — mini-epoch 중 일부에서만 optimizer step 발생. 표준 PPO는 매 mini-epoch마다 step. 의도적 설계인지 확인 필요.

10. **`_BatchScheduler` 하드코딩** (`serving/handlers.py:75, 108-113`) — max_batch_size=8, window=50ms. Config의 `serving.max_batch_size` (기본 1)와 불일치.

### 설계 문서 내

1. **04-foundation: SettingsFactory "3개 파이프라인"** — 실제 4개 메서드(for_training, for_inference, from_artifact, for_estimation). "3개"는 for_inference/from_artifact를 같은 카테고리로 묶은 것이나 불명확.

2. **05-model: QLoRA + head 상호작용** — "head 설정이 있어도 QLoRA 경로에서는 무시된다"고 하면서, 이것이 검증 단계에서 차단(error)인지 조용히 무시(warning)인지 미정의. 현재 코드는 조용히 무시.

3. **02-schema vs 03-examples: CLIP loss 생략** — loss 생략 규칙("HuggingFace AutoModelFor*, YOLO 등 training_step 구현 모델")에 CLIP 미포함. CLIP 시나리오에서 loss 생략 여부 불명확.

4. **07-training: total_steps 계산** — `warmup_ratio`로 warmup_steps 산출 시 "total_steps"가 무엇인지 미정의. epoch 기반 학습에서 max_steps 미명시 시 산출 로직 누락.

### AGENT.md (이전 감사에서 발견, 여전히 유효)
